三、漏洞与 Bug 风险修复结果

H1：实盘下单可能绕过 idempotency_key 幂等校验

已修复 资金/重复下单风险

位置：backend/account_trading/router.py:90-93、backend/account_trading/router.py:543-643

现象：OrderRequest 只在请求体 mode == "live" 时要求 idempotency_key。但实际下单引擎是根据 account_id 查出的账户类型决定的。如果调用方传入一个实盘账户 account_id，同时把 mode 写成 paper 或不一致，就可能进入实盘路径但没有幂等键。

影响：网络重试、前端重复点击、接口重放时，实盘订单缺少幂等保护，存在重复委托风险。

修复结果：已在账户解析后增加二次校验：请求 mode 必须与账户类型一致；只要账户类型为 live，就强制要求 idempotency_key，否则返回 400。

H2：模拟盘买入挂单未冻结资金，可能超额提交委托

已修复 模拟盘资金约束失真

位置：backend/account_trading/repository.py:1069-1234、backend/account_trading/repository.py:1358-1404

现象：模拟卖出挂单会冻结持仓数量，但买入挂单如果状态为 submitted，当前逻辑没有冻结对应现金；余额快照中的 available_cash 与 frozen_cash 也始终等于现金全额和 0。

影响：用户可以连续提交多个未成交买入挂单，总金额超过可用资金，导致模拟盘资金约束与真实交易不一致，复盘结果也会被污染。

修复结果：已增加模拟买入挂单冻结资金计算，余额快照会写入 available_cash = cash_balance - frozen_cash；部分成交后的剩余买单也会继续冻结资金，模拟盘重复请求会按 idempotency_key 返回已有订单。

M1：复盘区间内卖出持仓，如果买入发生在区间外，盈亏可能缺失

已修复 收益口径偏差

位置：backend/review_analysis/repository.py:321-359

现象：复盘使用区间内成交 FIFO 计算卖出盈亏。如果某只股票是在复盘开始日期前买入、区间内卖出，FIFO 队列没有开仓成本，卖出成交的 pnl 可能为空。

影响：阶段复盘、月度复盘、跨周期持仓复盘会低估或漏算已实现盈亏。

修复结果：复盘生成时会加载截至结束日期的历史成交用于成本计算，再只保留当前复盘区间内成交进入报告展示与统计，跨周期卖出的 FIFO 成本不再丢失。

M2：无资金快照时，复盘净值曲线只按现金变动计算，忽略持仓市值

已修复 净值/回撤失真

位置：backend/review_analysis/repository.py:361-412

现象：当账户没有余额快照时，系统用成交流水推导现金曲线，market_value 固定为 0。

影响：买入后现金下降会被误认为净值下降，未卖出的浮盈/浮亏不会体现在曲线中，最大回撤和累计收益可能明显错误。

修复结果：无资金快照时的 fallback 曲线已改为根据历史成交推导现金与持仓数量，并使用成交价估算持仓市值，避免把持仓市值固定为 0。

M3：未传 session_id 时默认取全局最新复盘会话，缺少账户隔离

已修复 多账户数据串读

位置：backend/review_analysis/repository.py:246-255

现象：_get_session(None) 会按 ID 倒序取全局最新会话，不按账户、用户或模块上下文过滤。

影响：多个账户连续生成复盘后，未指定 session_id 的接口可能返回另一个账户的报告。

修复结果：报告、交易明细、建议、净值曲线、回撤曲线接口均已强制要求 session_id，避免默认取全局最新会话造成多账户串读。

M4：复盘建议按 severity 字符串倒序排序，严重级别顺序不可靠

已修复 展示优先级错误

位置：backend/review_analysis/repository.py:202-209

现象：当前使用 severity.desc() 字符串排序。按字典序排序时，medium、low、high 的先后并不等于业务优先级。

影响：高风险建议可能没有排在最前面，影响使用者判断修复顺序。

修复结果：已使用 SQL case 排序：high = 3、medium = 2、low = 1，高严重级别会优先展示。

M5：订单查询页面存在但路由没有使用

已修复 前端入口不一致

位置：frontend/src/App.tsx:34-35、frontend/src/account-trading/pages/OrderQuery.tsx

现象：/account/orders 当前仍指向 TradingDeskPage，而独立的 OrderQuery 页面没有注册到路由。

影响：用户或其他模块按菜单/文档访问订单查询时，会进入交易台而不是订单查询页，容易造成页面职责混乱。

修复结果：OrderQueryPage 已注册到前端路由，/account/orders 现在指向独立订单查询页面。

L1：模拟盘 idempotency_key 只保存，不做重复请求拦截

已修复 幂等一致性

位置：backend/account_trading/repository.py:1104-1129

现象：模拟订单记录会保存 idempotency_key，但未见根据该字段查询已有订单并直接返回的逻辑。

影响：模拟盘重试时也会创建重复订单，虽然不涉及真实资金，但会污染复盘与测试数据。

修复结果：模拟盘下单已根据同账户 idempotency_key 查找已有订单，并直接返回已有委托、成交、余额和持仓结果。
